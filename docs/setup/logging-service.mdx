# Logging Service

By default, the PUDA CLI uses SQLite for local storage. This is enough for a local workstation, quick experiments, and single-user development.

For backups, persistent storage, and a more reliable record of the whole runtime environment, run a dedicated logging service. This service should run 24/7 on an always-on machine, preferably in a Docker container managed by Docker Compose.

The recommended template is [PUDAP/puda-logger](https://github.com/PUDAP/puda-logger), which listens to PUDA response streams from NATS and writes command responses to a database. The same pattern can be extended to log telemetry, events, alerts, or any other PUDA subjects described in the [communication layer](../overview/communication).

## Choose a database

Use PostgreSQL when you want a traditional relational database for command history, experiment records, joins, backups, and operational reporting.

Use InfluxDB when you want a time-series database for high-volume telemetry, sensor readings, and time-window queries.

The current `puda-logger` template includes a PostgreSQL logger under `nats/`. Use it as the reference implementation for logging from NATS and writing to a database. If you use InfluxDB, keep the same NATS subscription pattern and replace the database client with an InfluxDB writer.

## Run the PostgreSQL logger

Clone the logger template on an always-on machine that can reach your NATS server. This can be the same machine that runs NATS, or another reliable machine on the same network:

```bash
git clone https://github.com/PUDAP/puda-logger.git
cd puda-logger/nats
```

Create a local environment file:

```bash
cp .env.example .env
```

Set `NATS_SERVERS` to the same NATS addresses used by your PUDA CLI and edge services:

```env title=".env"
NATS_SERVERS=nats://<nats-server-ip>:4222
POSTGRES_HOST=postgres
POSTGRES_PORT=5432
POSTGRES_DB=puda
POSTGRES_USER=postgres
POSTGRES_PASSWORD=<change-this-password>
```

For a NATS cluster, provide all servers:

```env title=".env"
NATS_SERVERS=nats://nats1:4222,nats://nats2:4222,nats://nats3:4222
```

Start the logger and its PostgreSQL database as Docker containers:

```bash
docker compose up -d
```

The Compose template starts both services, restarts them automatically unless stopped, and stores PostgreSQL data in a Docker volume, so command logs remain available across container restarts.

## What the logger records

The logger subscribes to PUDA command response streams:

```text
puda.*.cmd.response.queue
puda.*.cmd.response.immediate
```

It writes those responses to the database so you have a durable backup of machine command results. The logger listens to responses, not command requests, so it records what machines returned after a command was handled.

You can also adapt the logger to subscribe to other subjects, such as `puda.*.tlm.>` for telemetry, `puda.*.evt.>` for events and alerts, or `puda.>` for a full namespace archive.

## Database writer template

Use the logger template as the pattern for any database backend:

```python title="logger.py"
class LoggerService(StreamSubscriber):
    async def on_start(self):
        await self.subscribe(
            "RESPONSE_QUEUE",
            "puda.*.cmd.response.queue",
            durable="logger_resp_queue",
        )
        await self.subscribe(
            "RESPONSE_IMMEDIATE",
            "puda.*.cmd.response.immediate",
            durable="logger_resp_immediate",
        )

    async def handle_message(self, msg, stream: str, subject: str):
        payload = msg.data.decode("utf-8")
        response = NATSMessage.model_validate_json(payload)

        if stream == "RESPONSE_QUEUE":
            command_type = "queue"
        elif stream == "RESPONSE_IMMEDIATE":
            command_type = "immediate"
        else:
            command_type = "unknown"

        self.db_client.insert_command_log(
            message=response,
            command_type=command_type,
        )
        await msg.ack()
```

For PostgreSQL, `db_client.insert_command_log(...)` should insert a row into a relational table. For InfluxDB, use the same parsed `response` object to write points with tags such as `machine_id`, `run_id`, `command_id`, and `command_type`.

## Verify logging

After the logger is running, send a command to a PUDA machine:

```bash
puda machine home <machine_id>
```

Then inspect the logger container:

```bash
docker compose logs -f logger
```

You should see the logger connected to NATS and PostgreSQL, then processing command responses as machines return results.
