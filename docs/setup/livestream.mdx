# Livestream

Use this setup when a PUDA machine host has one or more USB cameras or webcams attached and you want live video available over RTSP, RTMP, HLS, or WebRTC.

This guide assumes:

- Docker Engine and Docker Compose are installed.
- A camera or webcam is plugged into the host.
- The host can be reached by operators over the same private network or VPN used for PUDA.

## Find camera devices

On the livestream host, list video devices:

```bash
ls -l /dev/video*
```

If the machine has `v4l-utils` installed, inspect the cameras:

```bash
v4l2-ctl --list-devices
```

Choose the device paths to expose, such as `/dev/video0` and `/dev/video2`.

## Clone the livestream repo

Clone the livestream repo on the machine host:

```bash
git clone https://github.com/pudap/livestream.git
cd livestream
```

## Configure the livestream

Only the relevant file excerpts are shown below.

Set the host address in `.env`. Use an IP or hostname that viewers on your network or VPN can reach — for example a LAN address, VPN address, or public DNS name:

```env title=".env"
# Host address advertised to clients. Must be reachable from viewer machines.
HOST_ADDRESS=100.64.0.10
```

Define the cameras in `streams.conf`. Each non-blank, non-comment line defines one stream: the device path on the host and the public stream name, separated by whitespace.

```conf title="streams.conf"
# device path    stream name
/dev/video0      cam0
/dev/video2      cam1
```

In `compose.yml`, expose the same host video devices to the `ffmpeg` container:

```yaml title="compose.yml"
services:
  ffmpeg:
    devices:
      - /dev/video0:/dev/video0
      - /dev/video2:/dev/video2
```

The `devices:` entries make the host cameras visible inside the ffmpeg container. Add one line for each local `/dev/video*` camera listed in `streams.conf`, and remove entries for cameras you are not using.

If a port is already in use on the host, change the left-hand side of the matching `ports` entry in `compose.yml`.

To add or remove a camera later, edit `streams.conf`, update the `devices:` entries in `compose.yml`, and run:

```bash
docker compose restart ffmpeg
```

## Start the livestream

```bash
docker compose up -d
```

Check that both services are running:

```bash
docker compose ps
```

View logs if a stream does not appear:

```bash
docker compose logs -f mediamtx ffmpeg
```

## Stream URLs

For each stream name in `streams.conf`, replace `STREAM_NAME` and `100.64.0.10` with your stream name and the `HOST_ADDRESS` from `.env`:

| Protocol | URL |
| --- | --- |
| RTSP | `rtsp://100.64.0.10:8554/STREAM_NAME` |
| RTMP | `rtmp://100.64.0.10:1935/STREAM_NAME` |
| HLS | `http://100.64.0.10:8888/STREAM_NAME/` |
| WebRTC | `http://100.64.0.10:8889/STREAM_NAME/` |

With the example `streams.conf`, `cam0` and `cam1` are available at those URLs.

## Verify with ffprobe

From another machine that can reach the livestream host, verify a stream:

```bash
ffprobe rtsp://100.64.0.10:8554/cam0
```

For browser viewing, open the HLS or WebRTC URL.
